## ‚ö†Ô∏è What doesn‚Äôt work well with wikilinks on GitHub

Wiki links are *not* part of the standard GitHub-flavored Markdown. GitHub‚Äôs rendering doesn‚Äôt recognize `[[file]]` links (unless you convert them somehow) in Markdown files in the repo. So:

* If you use `[[file]]` in your README.md or in docs in your repo, GitHub itself won‚Äôt turn that into a clickable link to the file.
* That means if someone browses the repo on github.com, the link may just show up literally, or broken/unlinked.
* Tools which parse GitHub Markdown (CI, static site generators, search tools) may or may not handle wiki-links unless configured. So you lose portability unless all parts of your toolchain support them.

## ‚úÖ What works better (relative/absolute Markdown links)

Using standard Markdown links (e.g. `[Link Text](./file.md)` or `[Link Text](/file.md)` or `[Link Text](../file.md)`) has these advantages:

* GitHub supports them out of the box. Files in the repo can link to other files.
* If you move or rename files, as long as you update paths, things stay valid.
* Tools, editors, CI, static site generators generally understand them.
* More portable in the long run.

Downsides:

* They can get verbose.
* If moving files or reorganizing directories, you‚Äôve got to update many links.
* Relative paths can feel brittle.

## üõ† Practical suggestions / compromises

Here are a few paths you might take (leaning practical, not ideal):

1. **Use both formats**

   * Use wiki-links in drafts or notes (or where the AI uses them).
   * But alongside, put a standard Markdown link for the same target. Maybe in comments or secondary lines.
   * Or have a build step / CI check that converts wiki-links to Markdown links or verifies them.

2. **Adopt a custom parser / preprocessor**

   * If you control the tooling, you can write a small script that replaces `[[FileName]]` with `[FileName](./path/to/FileName.md)` based on a mapping (e.g. search index).
   * The AI can also use that mapping to resolve wikilinks internally.

3. **Stick with Markdown links in repo, but use wiki-link style for human readability**

   * E.g. you write `[[file]]` in your editor, but commit/link‚Äêcheck with `[file](path/to/file.md)`.
   * Your editor snippets or VSCode shortcuts might help.

4. **Structure repo to reduce path pain**

   * Keep many of your files in shallow directories so that relative paths are simpler.
   * Use consistent naming so that names are unique, helping both AI matching and human link creation.
